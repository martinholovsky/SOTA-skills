# SOTA-skills — Adversarial Repository Audit, 2026-07-10

Audit of the library at **v1.12.1** (HEAD `b188aa7`). Method: 13 fan-out
auditors across four lenses (decisions, code quality, security, strategy) +
an adversarial refutation pass that independently re-reproduced every non-trivial
finding; then orchestrator spot-verification of every headline finding against
a primary source or reproduced command. Every claim below cites evidence
produced **this run** — not the repo's own prose. Prior audit:
[AUDIT-2026-07-01.md](AUDIT-2026-07-01.md).

## Executive summary

**Overall health: STRONG.** The product — 254 rules files + 41 `SKILL.md`
read by AI agents — is factually accurate and free of dangerous advice under
adversarial sampling. All 6 CI invariants pass live (exit 0); supply-chain
pins are genuine; ~150 rot-prone claims (CVEs, RFCs, GA states, standards)
were sampled across every domain cluster and verified against primary sources,
with only a handful of small factual errors. Both HIGH findings from the
2026-07-01 audit are demonstrably remediated.

The real weaknesses are **strategic, not defects in shipped content**:

1. **No content-accuracy gate exists.** CI enforces *structure* (line count,
   headings, counts, version lockstep) but nothing checks whether a skill's
   claims are *true*. Accuracy rests entirely on a manual sweep behind a
   12-month freshness window — while the library's own measured drift is far
   faster (the 2026-07-08 sweep found 65 gaps in ~3 weeks). **HIGH.**
2. **The product's core efficacy claim is untested.** No eval harness has ever
   existed, despite `sota-llm-engineering` itself mandating golden-set evals.
   Whether loading a skill actually changes agent output is unmeasured. **HIGH.**
3. **Coverage expansion is an exhausted lever.** 12 releases / 41 skills in 23
   days against 4 stars, 2 forks, 1 issue ever. Effort is mis-aimed. **MEDIUM.**

Plus a tail of **low-severity content and script defects** (below) worth a
cleanup pass but none shipping-critical.

**Refutation outcome:** of 11 non-trivial findings sent to adversarial
verifiers, **11 confirmed, 0 refuted, 1 severity-downgraded** — i.e. the
finders did not inflate. 59 low/info observations passed through.

### Lens 2 — Code quality & architecture

- **Q-MED-1 (Medium):** `check-invariants.sh` check 2 (Audit-checklist-last)
  **bypasses a fenced heading.** Repro this run: a rules file whose only
  `## Audit checklist` sits inside a ```` ```markdown ```` fence passes.
  The 2026-07-01 fix was half-applied; the script comment overclaims.
  *Fix:* track fence state in awk — `awk '/^(```|~~~)/{f=!f} !f&&/^## /{h=$0}
  END{exit h!~/^## Audit checklist/}'`.
- **Q-MED-2 (Medium):** router **Library map omits `sota-confidential-computing`**
  (the 41st skill). Repro: `comm` of map bullets vs `skills/` dir → exactly
  `sota-confidential-computing` missing (`skills/sota/SKILL.md:229-347`). Two
  more bullets are stale: `sota-testing` lists 01-07 (dir has 09),
  `sota-docs-workflow` lists 01-04 (dir has 05). Routing table + per-skill
  indexes are correct, so routing still works — only the map overview is wrong.
  *Fix:* add the missing bullet; refresh the two stale ones.
- **Q-MED-3 (Medium):** structural checks the 2026-07-01 audit ran by hand
  (rules-index ↔ file existence, cross-ref resolution) were **never automated**
  — no such logic in `check-invariants.sh`. The Q-MED-2 drift is exactly what
  an automated check would have caught. *Fix:* add invariant 7 (below).
- **Q-MED-4 (Medium):** `install.sh` — **declining a routing prompt aborts the
  installer with exit 1**, silently skipping pre-commit setup + final
  instructions. Repro under a pty: `printf 'y\nn\n' | script -q … install.sh`
  → exit 1, output truncated. *Fix:* make declines `return 0`.
- **Q-MED-5 (Medium):** `install.sh:332` **clobbers a real profile file** in
  `~/.claude/profiles` via `ln -sfn` with no `-e`/`-L` check, no backup, no
  prompt — contradicting the script's own ask-first/backup-first contract used
  elsewhere. Repro: real file overwritten by symlink, content lost. *Fix:*
  `backup()` + `ask_yn` before linking, matching `setup_claude_md`.
- **Q-LOW cluster (quality):** check 4 under-counts folded descriptions with
  over-indented continuation lines (1201-char desc passed); check 2 accepts
  trailing text after the heading (`## Audit checklists are cool…`); check 6
  `set -e`-aborts on a tracked-but-deleted file; check 5 semver guard misses

  `gen-agents-md.sh`/`init-gates.sh` don't validate marker *order* (reversed

- **Q-LOW content pins (No-pins policy violations, quality):** `sota-rust`

### Lens 3 — Security posture

- **S-MED-1 (Medium):** `check-invariants.sh` check 3 **fails open in CI if
  `SOTA_DENYLIST` is ever deleted/renamed** — `ci.yml:30-32` injects the secret
  with no non-empty assertion; the script falls through to "generic checks only"
  and stays green. Repro: fresh clone, no secret → "private denylist
  unavailable" then `ok`. *Fix:* in `ci.yml`, fail (or `::warning::`) on push
  to main / non-fork PRs when the secret is empty.
- **S-MED-2 (Medium):** `sota-kubernetes/rules/01:209-210,228` states the 2026
  ingress-nginx CVE wave is **"unpatched"** — factually wrong. Verified this
  run: **1.13.9/1.14.5/1.15.1 are the final releases that DID patch** those
  CVEs (incl. config-injection/RCE CVE-2026-4342). The real risk is that CVEs
  found *after* the March-2026 EOL get no fix. *Fix:* reword — "patched in the
  final releases (≥1.13.9/1.14.5/1.15.1); post-EOL, any newly-found CVE is
  unpatched."
- **S-LOW cluster (security):** `SOTA_DENYLIST` is exfiltratable by any
  same-repo PR branch (secret exposed to PR-controlled `run` step) — internal
  names, low value, but real; `.gitleaks.toml` disabling `generic-api-key`
  leaves a documented detection gap (planted 40-char key not caught this run);
  ImageMagick `convert` GOOD examples pass attacker filenames where `--`
  doesn't stop `@`/`|` MSL abuse (`sota-python/rules/05:40`,
  `sota-sandboxing/rules/04:154`); denylist patterns pass through `grep` argv
  (process-list visible). All low.
- **Dangerous-advice sweep: CLEAN.** All 157 embedded audit greps executed;
  every risky pattern (TLS-skip, weak crypto, pickle, `shell=True`, `curl|bash`,
  privileged containers) is correctly framed as a BAD example / denylist entry /
  audit grep. No fail-open guidance, no copy-pasteable secrets.

### Lens 4 — Strategy & roadmap

- **STRAT-HIGH-1 (High):** no automated content-accuracy assurance; the annual
  window contradicts the measured drift rate (see Exec summary). *Fix:* rolling
  per-skill sweep (~3-4 skills/week ≈ full library/quarter); cut
  `--window-months` to 3-4; document the sweep-agent procedure in-repo.
- **STRAT-HIGH-2 (High):** no eval harness ever explored — the efficacy claim
  is untested. *Fix:* prototype a golden-set eval for 2-3 skills (e.g.
  code-security: 10 vulnerable snippets an audit must flag; router: 10 prompts
  → expected skill loadout) and measure with-vs-without.
- **STRAT-MED-1 (Medium):** coverage expansion is exhausted vs near-zero
  adoption (gh API this run: 4 stars / 2 forks / 1 issue). *Fix:* pause net-new
  skills a cycle; invest in distribution (marketplace visibility, a published
  before/after audit demo) and the two HIGH items.

## Unexplored ideas (promising, never tried)

1. **Content-accuracy CI gate** — a scheduled job that re-verifies a rotating
   subset of claims against primary sources and files issues on drift. The
   library preaches this; it doesn't practice it.
2. **Eval harness** — measure that skills actually improve agent output
   (golden sets). Turns the efficacy claim from asserted to demonstrated.
3. **Distribution / worked demos** — a public before/after audit run; the
   adoption signal (4 stars) says the bottleneck is visibility, not coverage.
4. **Badge → verifiable-audit** — link the "Built with SOTA Skills" badge to a
   committed audit report + commit SHA (idea logged, never built).
